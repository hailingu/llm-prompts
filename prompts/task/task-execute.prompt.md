Execute exactly one task: `${input:taskId}`.

## Inputs

- `${input:taskId}` (required): e.g. `Task-1.2`
- `${input:taskListPath}` (required): either
  - task-list file path, e.g. `docs/tasks/user-auth/task-list.md`
  - plan directory path, e.g. `docs/tasks/user-auth/`
- `${input:dryRun}` (optional boolean): simulate only, no file edits, no command execution

## File Resolution Rules

1. If `${input:taskListPath}` is a file, it must be `task-list.md`.
2. If `${input:taskListPath}` is a directory, use `{dir}/task-list.md`.
3. If resolved `task-list.md` does not exist, fail with `status: "error"`.

Task source of truth is always `task-list.md`.

## Contract Expected from `plan-breakdown-to-task`

The target task entry must include:

- Status line: `- Status: <status> (<中文>)`
- Checklist section with markdown checkboxes
- `Execution Parameters` YAML block

Required execution keys:
- `taskId`
- `acceptanceCriteria`
- `workspacePath`
- `timeoutMinutes`

## Output Requirements

- Update task status in Simplified Chinese and canonical status key.
- Update checklist from `[ ]` to `[x]` only after successful verification.
- Write execution evidence (commands run, test summary, failure details).
- Optional: if `{taskId}.md` exists, keep its status/checklist aligned with `task-list.md`.

## Status Flow

| Status | Chinese | Meaning |
| ------ | ------- | ------- |
| `pending` | 待执行 | initial |
| `in_progress` | 执行中 | implementation started |
| `testing` | 测试中 | verification running |
| `testing_failed` | 测试失败 | tests/quality checks failed |
| `completed` | 已完成 | all criteria passed |
| `failed` | 失败 | execution/validation/runtime error |

## dryRun Mode

When `${input:dryRun}` is true:

- Do not edit files.
- Do not execute commands.
- Return a simulated action plan and expected status transition.
- Use result status: `dry_run_completed` (模拟完成) in response only.

## Execution Workflow

1. Locate and parse task entry from `task-list.md` by `${input:taskId}`.
2. Validate required schema and required keys.
3. Validate dependencies are `completed`.
4. Update status to `in_progress`.
5. Run `runCommands` (if any) under `workspacePath`.
6. Update status to `testing`.
7. Run `testCommands` within `timeoutMinutes`.
8. Validate `acceptanceCriteria` evidence.
9. If all pass: set `completed`, mark checklist `[x]`.
10. If any fail: set `testing_failed` (or `failed` for runtime/validation errors), record details.

## Error Handling (Fail Fast)

No automatic retry.

| Scenario | Status | Action |
| -------- | ------ | ------ |
| Task not found | `failed` | return available task ids |
| Missing required schema | `failed` | return missing fields |
| Missing required execution keys | `failed` | return missing keys |
| Dependency not completed | `failed` | return blocking dependencies |
| Timeout | `failed` | cancel and record partial progress |
| Tests/quality gates failed | `testing_failed` | record failing command/evidence |

Error response format:

```json
{
  "status": "error",
  "taskId": "Task-1.2",
  "missing": ["acceptanceCriteria"],
  "message": {
    "zh": "缺少必填字段: acceptanceCriteria",
    "en": "Missing required field: acceptanceCriteria"
  }
}
```

## Hard Constraints

- Execute only the specified `taskId`.
- Do not modify unrelated tasks.
- Do not guess parameters absent from document.
- Do not output summary-only responses without state updates.
